# 数据备份与恢复

## 已确认决策（grill-me）
1. **场景**：灾难恢复 + 迁移 + 手动快照回滚 + 数据导出（全选）
2. **格式**：DB 文件副本（灾难恢复/迁移/快照）+ JSON 逻辑导出（分析）两者都要
3. **存放**：服务器本地目录 + UI 可下载 + 可上传恢复
4. **自动备份**：复用现有 node-cron scheduler，加 backup 任务（默认每天凌晨，Admin UI 可改周期）
5. **保留策略**：后台可配（默认保留最近 7 份，超出自动删最旧）
6. **恢复保护**：恢复前自动备份当前 DB（"恢复前快照"），再覆盖
7. **JSON 导出范围**：仅业务数据（Floor/Zone/Workstation/Person/NewsItem/Resource/Category）；不含 User/ApiKey/SyncLog/Setting
8. **DB 文件备份**：完整 SQLite 副本（含所有表），依赖文件系统权限保护

## 背景
- 单 SQLite 文件 `prisma/db.sqlite`（172KB，11 表），单进程部署
- `better-sqlite3` 驱动适配器，提供在线安全备份 API（`.backup()`）
- 无任何现有备份/导出路由
- scheduler 已有 3 个任务（sync-members/attendance/publish-news），模式成熟

---

## 一、备份目录与工具（`lib/backup.ts`，新建）
- 目录：`prisma/backups/`（gitignore）
- **`createBackup(trigger: 'auto'|'manual'|'pre-restore')`**：
  - 用 better-sqlite3 的 `db.backup(toPath)` 在线安全复制（不断开 Prisma 连接）
  - 文件名：`backup-<timestamp>-<trigger>.sqlite`（如 `backup-20260720T030000-auto.sqlite`）
  - 返回 `{ filename, sizeKb, trigger, createdAt }`
- **`listBackups()`**：扫描目录，按时间倒序返回元信息
- **`deleteBackup(filename)`**：删除单个
- **`pruneBackups(keepN)`**：保留最近 N 份（按 mtime），删超出
- **`restoreBackup(filename)`**：先 `createBackup('pre-restore')`，再用 `db.backup()` 反向覆盖主 DB（better-sqlite3 backup 支持双向：`new Database(main).backup(fromPath)` 或文件复制）
  - 覆盖后需让 Prisma 重连（见下"恢复的连接处理"）
- 保留份数从 Setting 表读（key=`backup.keep`，默认 7）

### 恢复的连接处理
better-sqlite3 的 `.backup()` 在线操作，不需要断开 Prisma。但恢复后 Prisma 可能有缓存——恢复后调 `prisma.$reset()`?Prisma 没有 public reset。实际：SQLite 备份 API 写入的是同一个文件，better-sqlite3 句柄共享，Prisma 透过 adapter 读时会看到新数据。验证 E2E 时确认；若需，让用户刷新页面（SWR 重新拉）。

## 二、调度任务（`lib/scheduler.ts`）
- `JOB_DEFS` 加一个 `backup` 任务：`run: () => { createBackup('auto'); pruneBackups(readKeep()) }`
- `cron-presets.ts` 加 `dailyMidnight`（已有）作默认（`0 0 * * *`）
- `CRON_DEFAULTS` 加 `'backup': '0 0 * * *'`
- scheduling-panel 的 JOBS 加 backup 行（预设 + 自定义 cron + enable）
- SyncLog 记录（job='backup'，source='cron'/'manual'）

## 三、备份 API（`app/api/backup/`，新建，全 requireAdmin）
- `GET /api/backup` — 列出备份（listBackups）
- `POST /api/backup` — 手动创建（createBackup('manual') + prune）
- `POST /api/backup/restore` — body `{ filename }` → restoreBackup（含预备份）
- `DELETE /api/backup?filename=X` — 删单个
- `GET /api/backup/download?filename=X` — 文件下载（`Content-Disposition: attachment`）

## 四、JSON 导出 API（`app/api/export/route.ts`，新建，requireAdmin）
- `GET /api/export` — 导出业务表为 JSON：
  ```json
  { "exportedAt": "...", "tables": { "floor": [...], "zone": [...], ... } }
  ```
- Content-Disposition 触发下载，文件名 `idrl-export-<date>.json`
- 仅：Floor/Zone/Workstation/Person/NewsItem/Resource/Category（7 表）
- 用 `toFloor`/`toPerson` 等现有 serialize

## 五、上传恢复（`POST /api/backup/upload`）
- 接受 FormData（一个 .sqlite 文件）
- 保存到 `prisma/backups/upload-<timestamp>.sqlite`
- 走 restoreBackup 流程（先预备份，再覆盖）
- 校验：文件是有效 SQLite（打开试读 `_prisma_migrations` 表）

## 六、保留配置（Setting）
- key=`backup.keep`，默认 `"7"`
- Admin UI 可改（scheduling-panel 或单独输入框）
- backup 任务执行时读

## 七、Admin UI（`app/dashboard/admin/`）
新 tab **「备份」**（`components/admin/backup-panel.tsx`）：
- **手动备份**按钮（POST /api/backup）+ 结果提示
- **备份列表**表格：文件名 / 时间 / 触发方式（auto/manual/pre-restore）/ 大小 / 操作（下载 / 恢复[确认] / 删除）
- **上传恢复**：文件选择 + 上传按钮（FormData），提示"恢复前会自动备份当前数据"
- **JSON 导出**按钮（GET /api/export → 下载）
- **保留份数**输入框（保存到 Setting）
- 恢复操作：二次确认弹窗（"将覆盖所有当前数据"）

admin/page.tsx 加 `<TabsTrigger value="backup">` + TabsContent。

## 不在范围
- 异地/云存储备份（本地目录够单机用）
- 增量备份（全量 SQLite 文件小，172KB，无需增量）
- 备份加密（依赖文件系统权限）
- 按表勾选导出（固定业务 7 表）
- 不做"最后一个 admin 保护"之外的恢复额外校验（已有预备份兜底）
